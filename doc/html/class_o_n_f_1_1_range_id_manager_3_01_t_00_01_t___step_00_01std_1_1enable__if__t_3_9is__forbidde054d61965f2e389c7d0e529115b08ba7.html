template<class T, class T_Step>
class ONF::RangeIdManager< T, T_Step, std::enable_if_t<!is_forbidden_types_combination< T, T_Step > > >

Диапазонный IdManager на основе IdRange.

RangeIdManager это IdManager который состоит из IdRange и IdContainer-в.
Так как в состав RangeIdManager входит IdRange и IdContainer, то на RangeIdManager распространияются те же правила что и на IdRange с IdContainer.
Прежде всего стоит ознакомиться с IdRange и IdContainer.

Особенность IdContainer в том что он может выдавать ID разными способами.
За то каким образом IdContainer будет выдавать ID отвечает IdIssuingMethod.
Но если в IdContainer данное перечисление отвечает только за порядок выдачи ID, то в RangeIdManager оно еще отвечае за поведение границ.

Стоит выделить два режима границ: динамический и статический.
При динамическом типе границ, эти самые границы могут как раздвигаться так и сдвигаться:

К другим особенностям динамического режима стоит отнести наличие так называемого "перепрыгивания", чего нету при статическом режиме, но о Перепрыгивании позже.
Так же стоит сказать о том что границы при динамическом режиме синхронизированы между собой.
Но это скорее естественный результат нежели намеренная особенность.
В чем суть. Границы как при динамическом так и при статическом режиме могут иметь два состояния: true и false.
В обоих режимах если граница имеет состояние true (граница включена), значит ID на котором она находится зарезервирован.
Если граница имеет состояние false (граница выключена), значит ID на котором она находится свободен.
Если в динамическом режиме освободить ID на который указывает одна из границ, то граница не выключется, как это произошло бы в статическом режиме, она просто сдвинется назад в сторону другой границы.
Если просле того как граница сдвинулась назад она попадает опять на свободный ID, движение назад продолжается.
Так будет до тех пор пока граница не окажется на зарезервированном ID или пока не столкнется с другой границей.
От сюда следует что в динамическом режиме границы всегда показывают на зарезервированный ID и всегда находятся во включенном состоянии, кроме случая когда мы не зарезервировали ни одного ID, тогда границы находятся в одной точке и обе выключены.

Как и было сказано, в статическом режиме границы не могут сдвигаться. Это сделано для того что бы RangeIdManager мог выдавать ID в порядке возрастания или убывания что невозможно в динамическом режиме.
Имеется ввиду что не всегда возможно выдавать освобожденные ID в порядке возрастания или убывания.
Это из-за того что при динамическом режиме, освобождая ID находящийся на границе, эта граница сдвигается, что может помешеть выдавать ID в нужном порядке.

Мы можем получать свободные ID с помощью getFreeId(). Этот метод просто раздвигает границы, но если в пределах диапазона есть свободный ID он вернет его.
Так же getFreeId() умеет восстанавливать диапазон. Об этом позже.
Но мы так же можем резервировать любой ID с помощью reserve(T, ReservationMethod).
reserve(T, ReservationMethod) может резервировать ID на границе, в пределах и за пределами диапазона.
В обоих режимах следующий за границей ID всегда свободен.
Если мы зарезервируем с помощью reserve(T, ReservationMethod) следующий за границей ID, граница рядом с которой находится этот ID передвинется на него.
После это граница проверит, зарезервирован ли следующий ID. Если да, она передвинется на него. И так будет до тех пор пока следующий ID не окажется пустым.

Так же хочется отметить еще одну особенность динамического режима.
При динамическом режиме границы стремятся находиться на своей территории.
О территориях можно узнать из описания IdRange::IdInfo::border.
При динамическом режиме возможна ситуация когда обе границы будут по одну сторону от стартового ID, что невозможно при статическом режиме.
Когда это произойдет, та граница которая оказалась не на своей территории начнет постепенно возвращаться на свою территорию. Это называется восстановление диапазона.
Граница будет считать что на своей территории когда окажется на стартовом ID.

По умолчанию обе границы выключены, находятся на ID равном 0 и с размером шага 1.
Лимит BorderRange::UpperBorder по умолчанию равен std::numeric_limits::max().
Лимит BorderRange::LowerBorder по умолчанию равен std::numeric_limits::lowest().
Метод выдачи ID по умолчанию равен IdIssuingMethod::Dynamic.
Так же по умолчанию установлен жесткий шаг.

Шаг не может быть равен 0. Если вы передадите в step 0, он автоматически станет равен 1.
Знак числа шага, передаваемого в step, не имеет значения, минус автоматически отбрасывается.

Аргументы
start- стартовый ID.
step- размер шага.

Возвращает ближайший к переданному ID, стандартный ID.

Лимит не распространяется на данный метод.
В id надо передать тот ID, ближайший стандартный ID к которому нужно найти, а в borderRange надо передать ту границу от которой будет вестить отсчет.
Т.е. искомый ID будет находиться между id и границей переданной в borderRange.
Данный метод основан на функции ONF::findNearestStdId(T, T, T).
Параметр target данной функции это параметр id данного метода, а параметр start это borderRange.

Параметр borderRange необходим из-за особенности динамического режима, а именно из-за того что обе границы могут оказаться по одну сторону от стартового ID.
Подробнее о данной особенности можно прочитать в описании данного класса.
Если бы не эта особенность динамического режима, borderRange можно было бы откинут и оставить только id.
В таком случае отсчет мог бы вестись от стартового ID.
Но из-за динамического режима это невозможно потому что возникнет такая проблема:

rangeIdManager.findNearestStdId(5.5); //result: 5

В первом случае все хорошо, ближайший стандартный к границе диапазона ID равен 5.
А вот во втором случае ошибка. Ближайшим стандартным ID к границе диапазона является 2, но метод возвратит 1.
В целом это не ошибка, но для данного класса это все же она. Потому был добавлен параметр borderRange. Он уточняет, от какой границы ведется отсчет.
В примере выше, в обоих случаях можно передавать в borderRange как BorderRange::UpperBorder, так и BorderRange::LowerBorder, результат будет один, и этот результат будет правильным.

◆ getFreeId()

template<class T , class T_Step >
std::optional<T> ONF::RangeIdManager< T, T_Step, std::enable_if_t<!is_forbidden_types_combination< T, T_Step > > >::getFreeId ( )

Возвращает свободный ID в соответствии с режимом выдачи ID заданным для данного объекта класса.

При первом вызове данного метода он возвращает стартовый ID, а при каждом последующем вызове будет возвращать ID в соответствии с размером и направлением шага.
Например, если Стартовый ID равен 3, а шаг равен -2, то при первом вызове данный метод вернет 3, затем 1, -1, -3, -5 и т.д.
Если в пределах диапазона появляются свободные ID, метод будет возвращать и их тоже, но это будет происходить в соответствии с заданным режимом выдачи ID.
Если однажды граница диапазона упрется в Лимит и свободных ID в пределах диапазона не будет, метод начнет возвращать std::nullopt.
Тут важно подчеркнуть тот момент что данный метод при обычных условиях работает с той границей которая соответствует направлению шага.
Если шаг с '-', данный метод будет работать с BorderRange::LowerBorder, а если с '+', то с BorderRange::UpperBorder.
Если например мы работает с BorderRange::UpperBorder и он упирается в Лимит, то данный метод начнет возвращать std::nullopt даже несмотря на то что BorderRange::LowerBorder еще не уперся в Лимит.

При динамическом режиме, данному методу свойственно востанавливать диапазон если это необходимо.
Порядок действий данного метода при динамическом режиме:

  1. Проверяет, не находится ли BorderRange::LowerBorder на территории BorderRange::UpperBorder. Если да, пытается восстановить. Если нет, переходит к пункту 2.
  2. Проверяет, не находится ли BorderRange::UpperBorder на территории BorderRange::LowerBorder. Если да, пытается восстановить. Если нет, переходит к пункту 3.
  3. Если в пределах диапазона есть свободный ID, возвращает его, иначе переходит к пункту 4.
  4. Если возможно передвинуть границу в соответствии с размером и направлением шага, возвращает ID на который была передвинута граница, иначе возвращает std::nullopt.

При статическом режиме выдачи ID равном IdIssuingMethod::Static_Ascending порядок действий данного метода:

  1. Проверяет, включена ли нижняя граница. Если нет, включит ее и вернет ее ID. Если включена, переходит к пункту 2.
  2. Если в пределах диапазона есть свободный ID, возвращает его, иначе переходит к пункту 3.
  3. Проверяет, включена ли верхняя граница. Если нет, включит ее и вернет ее ID. Если включена, переходит к пункту 4.
  4. Если возможно передвинуть границу в соответствии с размером и направлением шага, возвращает ID на который была передвинута граница, иначе возвращает std::nullopt.

При статическом режиме выдачи ID равном IdIssuingMethod::Static_Descending порядок действий данного метода:

  1. Проверяет, включена ли верхняя граница. Если нет, включит ее и вернет ее ID. Если включена, переходит к пункту 2.
  2. Если в пределах диапазона есть свободный ID, возвращает его, иначе переходит к пункту 3.
  3. Проверяет, включена ли нижняя граница. Если нет, включит ее и вернет ее ID. Если включена, переходит к пункту 4.
  4. Если возможно передвинуть границу в соответствии с размером и направлением шага, возвращает ID на который была передвинута граница, иначе возвращает std::nullopt.
Возвращает
Если удалось найти свободный ID, возвращает его, иначе возвращает std::nullopt.

◆ reserve()

template<class T , class T_Step >
bool ONF::RangeIdManager< T, T_Step, std::enable_if_t<!is_forbidden_types_combination< T, T_Step > > >::reserve ( T  id,
ReservationMethod  reservationMethod = ReservationMethod::NotSet 
)

Резервирует ID разными способами.

Данный метод имеет 2 параметра:

  • id - ID который нужно зарезервировать если это возможно и одновременно цель, от границы диапазона к которой образуется абстрактная область.
    Данная обласить может быть как зарезервирована, так и интерполирована.
  • reservationMethod - Метод резервации ID.

За метод резервации ID отвечает перечисление ReservationMethod.
О том как влияют его перечислители на процесс резервации ID можно вкрадце прочесть здесь:

  • ReservationMethod::NotSet
  • ReservationMethod::Interpolate
  • ReservationMethod::ReserveRange

А теперь более подробно об их работе. Далее поочередно в подробностях будет описан каждый перечислитель.
Перед использованием метода reserve(T, ReservationMethod) стоит ознакомится с каждым перечислителем что бы понимать все правила данного метода.
Так же стоит помнить что данный метод по сравнению с тем же getFreeId() является очень медленным.

Начнем с ReservationMethod::NotSet.
Данный перечислитель будучи переданный в метод reserve(T, ReservationMethod) говорит ему о том что мы просто хотим зарезервировать указанный ID если это возможно.

На метод reserve(T, ReservationMethod) расспространяются те же правила и обязанности что и на любой другой метод класса RangeIdManager.
В эти обязанности входит в том числе и обязанность передвинуть границу диапазона в случае если reserve(T, ReservationMethod) резервирует ID являющийся следующим за данной границей ID.
Подробнее об этом правиле в описании данного класса.
Так же, если в динамическом режиме границы диапазона находятся в одной точке будучи выключены, а мы попробуем зарезервировать следующий за одной из границ ID, произойдет перепрыгивание на данный ID.
В остальном данный перечислитель говорит просто зарезервировать указанный ID.

Далее идет ReservationMethod::ReserveRange.
Данный перечислитель говорит методу reserve(T, ReservationMethod) зарезервировать все ID от текущей границы до указанного ID.
При этом если при резервации промежуточных ID, если будет встречен уже зарезервированный ID, он просто пропускается. А вот все свободные ID резервируются.
Несмотря на то что среди промежуточных ID могут быть уже зарезервированные, ID переданный в id должен быть свободен.

Написать про то что отключенные границы так же резервируются и прочее.
